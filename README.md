# Семантичний і гібридний пошук наукових статей

Проєкт готує вибірку метаданих arXiv, створює нормалізовані ембеддинги
наукових статей за допомогою моделі `allenai/specter2_base`, завантажує їх у
Pinecone та демонструє семантичний, локальний, чанковий і гібридний пошук.

## Структура проєкту

- `scripts/01_prepare_data.py` — читає метадані arXiv і створює вибірку з
  10 000 статей у `data/arxiv_subset.parquet`;
- `scripts/02_embed.py` — генерує нормалізовані ембеддинги та зберігає їх у
  `embeddings/embeddings.npy`;
- `scripts/03_load_to_pinecone.py` — створює індекс `arxiv-papers` і
  завантажує в нього статті;
- `scripts/04_search.py` — виконує семантичний пошук із фільтрами та локально
  порівнює cosine, dot product і L2;
- `scripts/05_chunking.py` — порівнює fixed-size і semantic chunking,
  створюючи індекси `arxiv-chunks-fixed` і `arxiv-chunks-semantic`;
- `scripts/06_hybrid_search.py` — порівнює BM25, векторний пошук і гібридний
  пошук через Reciprocal Rank Fusion;
- `requirements.txt` — Python-залежності проєкту;
- `.env` — локальні змінні середовища, які не додаються до Git.

Каталоги `data/` та `embeddings/`, а також початковий датасет створюються або
додаються локально й виключені з Git через `.gitignore`.

## Налаштування оточення

### Передумови

Потрібні:

- Python 3.10 або новіший;
- обліковий запис Pinecone та API key;
- файл `arxiv-metadata-oai-snapshot.json` із набору метаданих arXiv,
  розміщений у корені репозиторію;
- доступ до інтернету для завантаження Python-залежностей, моделі
  `allenai/specter2_base` і роботи з Pinecone;
- достатньо вільного місця на диску: початковий JSON-файл має розмір у кілька
  гігабайтів.

Усі скрипти використовують відносні шляхи, тому їх потрібно запускати з
кореня репозиторію.

### Встановлення залежностей

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

Під час першого запуску скрипта генерації ембеддингів модель буде
завантажена з Hugging Face і збережена в локальному кеші.

### Налаштування змінних середовища

У корені проєкту потрібно створити файл `.env`:

```dotenv
PINECONE_API_KEY="<your-pinecone-api-key>"
```

Отримати ключ можна в консолі Pinecone. Файл `.env` доданий до `.gitignore`,
тому API key не потрапляє в Git. Скрипти, які працюють із Pinecone,
автоматично завантажують цю змінну через `python-dotenv`.

## Порядок запуску

Команди виконуються з кореня репозиторію в активованому віртуальному
середовищі:

```bash
# 1. Підготувати вибірку з 10 000 статей
python scripts/01_prepare_data.py

# 2. Створити ембеддинги
python scripts/02_embed.py

# 3. Створити індекс arxiv-papers і завантажити дані в Pinecone
python scripts/03_load_to_pinecone.py

# 4. Виконати семантичний пошук і порівняти метрики
python scripts/04_search.py

# 5. Створити чанкові індекси та порівняти стратегії chunking
python scripts/05_chunking.py

# 6. Порівняти BM25, векторний і гібридний пошук
python scripts/06_hybrid_search.py
```

Скрипти потрібно запускати саме в цьому порядку: наступні етапи використовують
файли або Pinecone-індекси, створені попередніми. Повторний запуск скриптів
завантаження оновлює вектори з тими самими ідентифікаторами.

### 1. Чим Pinecone відрізняється від Qdrant і Chroma за моделлю розгортання, ліцензією і продуктивністю? У якому сценарії ви б обрали кожен із них?

Pinecone - serverless рішення, тобто, VDB розгортається у cloud, не потребує налаштування інфраструктури, інфраструктуру налаштовується, мастабується, підтримується провайдером. Зручно для старту, або якщо немає виділеного infra engineer або devops (що типово для стартапів). Мінус - дані зберігаются не на контрольованих нами серверах, може порушувати  complience (особливо критичо для фінансового сектору), менше можливості тюнингувати низькорівневі параметри індексу. Вартість виростає при збільшенні даних, запитів. Найкраще підходить для стартапів або початкових тестувань концептів.

Qdrant - self hosted рішення (хоча є і cloud опція), що потребує додаткових operation витрат (зазвичай потребує щоб у команді був infra/devops engineer), так як рішення self hosted - повний контроль над даними, гнучкість налаштування параметрів індексу, реалізований гібридний пошук. Найкаще підходить для середніх і великих компаній, де proof of concept уже протестований і є достатньо ресурсів для підтримки інфраструктури.

Chroma - self histed рішення (є опція запускати як бібліотеку), найкраще підходить для локального тестування (моментальне тествання) і навчання, не призначена для високого навантаження чи великих обємів.

### 2. Чому для задачі пошуку по науковим текстам обрана модель specter2_base, а не універсальна all-MiniLM-L6-v2? Знайдіть картку моделі на HuggingFace і процитуйте, для яких задач вона навчена.

> SPECTER2 is the successor to SPECTER and is capable of generating task specific embeddings for scientific tasks when paired with adapters. This is the base model to be used along with the adapters. Given the combination of title and abstract of a scientific paper or a short texual query, the model can be used to generate effective embeddings to be used in downstream applications.

ll-MiniLM-L6-v2 - універсальна модель для коротких тестів, specter2_base - спеціалізована для наукових статтів, розуміє близкість статтів, коли автор статті посилається на іншу наукову статтю, що є типовим для наукових статтей. Тобто, all-MiniLM-L6-v2 працює в цілому добре для текстів, проте, specter2_base натренована саме на наукових статтях і краще розуміє не просто семантику, але посилання з однієї статті на іншу, що корисно при пошуці.

### 3. Що написано у картці моделі про рекомендовану метрику схожості? Чому це важливо при створенні індексу?

У самій картці не вказано рекомендовану метику схожості. Проте, у статті, яка згадується у картці "A Multi-Format Benchmark for Scientific Document
Representations" (https://www.semanticscholar.org/reader/0d68e244faa9a5944228b571d91bc7bed88083cd) вказується, що використовувалась косинусна схожість. 
Метрика порівняння залежить від моделі, тому що, модель при навчанні будує та опитмізує векторний простір ід конкретну метрику порівняння. Тому, якщо метрики для порівняння яка використовується не співпадає тій, з якою навчалась модель - ми отримаємо не вірний результат.

### 4. Поясніть, чому при використанні нормалізованих ембеддингів (одиничної довжини) косинусна схожість (cosine similarity) еквівалентна скалярному добутку (dot product)?

Косинусна схожість згідно формули це - скалярний добуток розділений на добуток норм векторів: 
$$\text{Cosine Similarity}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \|\mathbf{v}\|}$$
У випадку нормалізації векторів - усі довжини векторів дорівнюють одиниці, тому, при нормалізованих векторах формула спрощується до скалярного добутку.

## Завдання 3

### 1. Чи збігаються топ-5 для cosine і dot product і чому?
```
Top-5 Cosine: [378, 3350, 4115, 610, 3181]
Top-5 Dot Product: [378, 3350, 4115, 610, 3181]
Top-5 L2 Distance: [378, 3350, 4115, 610, 3181]
```
Так, усі топ-5 для різних метрик збігаються, тому що ми у завданні 2 нормалізували ембединги (`normalize_embeddings=True`), і формула косинусної схожості перетворилась у dot product.

### 2. Чи відрізняються результати для L2 і чому?

Для нормалізованих векторів існує пряма математична залежність між L2 відстанню  та скалярним добутком:
$$D_{L2}^2(\mathbf{u}, \mathbf{v}) = \|\mathbf{u}\|^2 + \|\mathbf{v}\|^2 - 2(\mathbf{u} \cdot \mathbf{v}) = 1 + 1 - 2(\mathbf{u} \cdot \mathbf{v}) = 2 - 2(\mathbf{u} \cdot \mathbf{v})$$
Найкоротша відстань по прямій ($L_2$) строго відповідає найменшому куту між векторами.

### 3. Що сталося б, якби ембеддинги не були нормалізовані?

Для косинусної схожості нічого б не змінилось, так як формула уже містить у собі нормалізацію векторів.
L2 сильно б змінився, так як вимірював б уже не різницю у напрямку, а геометричну відстань, і два вектори, які знаходяться на одній прямій, могли б показувати велику різницю через різні норми векторів (тобто, по косинусу вони б були б абсолютно схожі, але не по L2).
Dot product піднімав б у верх вектори з більшою довжиною (довші тексти), а короткі б "штрафував", тому більш релевантніші, але коротші тексти могли б бути по схожості нище, ніж менш релевантніші (по косинусу), але значно більші тексти.

## Завдання 4

### 1. Яка стратегія дає більш осмислені чанки?
Semantic Chunking дає більш осмислені чанки, тому що текст ділиться на чанки по реченням у випадку, коли сусідні речення починають між собою розходитись по косинусній відстанні, а у випадку fixed size chunk це ділення відбувається у будь-якому місці речення, розриваючи логічні речення і абзаци по різним чанкам.

### 2. Чи є випадки розрізаних речень і як це впливає на ембеддинги?

У стратегії Fixed size chunking випадки розрізаних речень трапляються постійно, так як алгоритм не зважає на розділові знаки, враховуются лише кількість слів.
У стратегії Semantic chunking  речення залишаються цілими, окрім  випадків, коли одне речення саме по собі перевищує ліміт максимально дозволеного чанку.

### 3. Як розмір overlap впливає на кількість чанків і покриття тексту?

Від overlap напряму залежить кількість чанків, чим більше це значення, тим більше буде чанків, так як вікно зміщення буде менше (залежить від розміру чанка і overlap). 
Перекриття тексту важливе, так як при розриві чанків може втрачатись семантика тексту, overlap покращує семантику на початку чанків.

## Завдання 5

### 1. Який метод дав кращий результат і чому?
Результат залежав від запиту (`query`). Для "BERT fine-tuning" (строга термінологія) та "Yann LeCun convolutional networks" (ім'я автора) найкращий резульат показав BM25. Для запиту "making computers understand human emotions from text" найкращий результат дав векторний пошук, тому що тут довільний текст, і BM25 шукав загальні (computers, text, human) слова які зустрічаються у багатьох статтях, а векторний пошук шукав саме семантичну схожість.
Гібридний метод в жодному екмперименті не давав кращий результат, але завжди мав у топі релевантні значення пошуку (проте менше релевантних результатів, а ніж метод який виграв у конкретному експерименті).

### 2. Чи є документи в топ-5 гібридного пошуку, яких немає в топ-5 окремих методів, і чому?

Так, такі випадки є. Причина полягає у формулі розрахунку балів RRF. Бали від обох методів сумують, і з параметром k=60, якщо документ має 6 ранг в обох методах, то результуючий бал буде вищий, а ніж у документа, який має 1 ранг в одному методі і взагалі не попав в топ другого метода. 

### 3. Як зміна параметра k в RRF впливає на видачу (наприклад, k=60 vs k=1)?

Параметр `k` контролює міру затухання (штраф) за падіння в ранзі. При `k=1` штраф за позицію величезний. Перше місце отримує `1/(1+1) = 0.5`, а друге — `1/(1+2) = 0.33`. Різниця між 1 та 2 місцем колосальна. Це призводить до того, що документ, який посів 1-ше місце в одному методі, майже гарантовано обійде документ, який посів 2-гі чи 3-ті місця в обох методах разом.
При `k=60` (стандартне значення): Різниця між першими рангами згладжується. Це покращує консенсус між моделями: системі стає важливіше, щоб документ згадувався в обох списках (нехай навіть на нижчих позиціях), ніж щоб він був абсолютним лідером лише в одному з них.

BM25 працює як класичний пошук у Google: бачить точне слово "LeCun" або "BERT" і відразу видає потрібну статтю на перше місце. Семантичний пошук у таких випадках часто галюцинує зайвого і замість конкретного автора підкидає схожі за темою статті інших людей.

Де виграв Семантичний пошук: На запиті "making computers understand human emotions from text". Тут немає розумних термінів, просто опис ідеї «своїми словами». Загальне правило: Якщо треба знайти конкретну назву, абревіатуру, код помилки чи прізвище — краще використовувати BM25. Якщо запит написаний простими словами, описує якусь абстрактну ідею чи проблему — семантичний пошук через ембеддинги однозначно показує кращі результати. Гібридний пошук через RRF  бере топ з обох методів і обєднує в збалансовану видачу.

### 2. Вплив розміру чанка. Що відбувається з якістю пошуку, якщо чанк занадто маленький (10–15 слів)? Якщо занадто великий (500+ слів)? Чи є оптимальний розмір або він залежить від задачі?

Якщо зробити чанки дуже маленьким (10–15 слів): Текст просто рветься на шматки, втрачаючи будь-який сенс. Модель векторищує окреме речення, не розуміючи загального контексту статті. На виході ми отримуємо купу «сміттєвих» векторів, і при пошуку система буде повертати випадковий результат, який просто збігся за парою слів.
Якщо зробити чанки величезними (500+ слів): Виникає інша проблема — усереднення змісту. Якщо добавити в один чанк кілька різних думок чи абзаців, модель спробує стиснути весь цей мікс в один вектор. У результаті важливі дрібні деталі просто розчиняться у загальному тексті, і знайти статтю за конкретним точковим фактом уже не вийде. 
Оптимальний розмір залежить від завдання. Розмір чанка дуже залежить від задачі, типу документа, embedding-моделі і того, як потім буде працювати retrieval у RAG. Головна ідея: чанк має бути достатньо малим, щоб точно відповідати на запит, але достатньо великим, щоб не втратити контекст.  Для прикладу, для коротких текстів (FAQ, короткі інструкції) зазвичай використовують 100–300 токенів, а для довгих текстів (статті, книги) 500–1200 токенів.

### 3. Невідповідна метрика. Що сталося б, якби ми створили індекс Pinecone з метрикою euclidean (L2), але використовували модель, яка повертає нормалізовані вектори? Обґрунтуйте відповідь математично: виведіть зв’язок між L2 і cosine для одиничних векторів.
Видача топ-5 буде абсолютно такою ж, як і для косинусної схожості.
Формулу Евклідової відстані через скалярний добуток, ми отримаємо:
$$\|\mathbf{u} - \mathbf{v}\|^2 = \|\mathbf{u}\|^2 + \|\mathbf{v}\|^2 - 2(\mathbf{u} \cdot \mathbf{v})$$
Якщо вектори нормалізовані, їхня геометрична довжина завжди дорівнює одиниці ($\|\mathbf{u}\|=1$ і $\|\mathbf{v}\|=1$).
$$\|\mathbf{u} - \mathbf{v}\|^2 = 1 + 1 - 2(\mathbf{u} \cdot \mathbf{v}) = 2 - 2(\mathbf{u} \cdot \mathbf{v})$$
Скалярний добуток ($\mathbf{u} \cdot \mathbf{v}$) для нормалізованих векторів — це і є чиста косинусна схожість. Виходить пряма залежність: що більша косинусна схожість між статтями, то меншою буде Евклідова відстань між ними.

### 4. Обмеження Pinecone Starter. З якими обмеженнями безкоштовного тіру ви зіткнулися (або могли б зіткнутися)? Як би ви вирішили задачу, якби датасет був не 10000, а 10 мільйонів статей?

На безкоштовному тарифі Pinecone Starter є жорсткі ліміти:
1. Обмеження метаданих: В один вектор не можна добавити метадані вагою понад 40 КБ (тому ми й обрізали `abstraction` до 500 символів, щоб не виходити за ліміти).
2. Регіони: Не можна вибрати сервер ближче до себе, безкоштовно дають лише один фіксований регіон на AWS. Не зміг обрати eu регіон (хоча знаходжусь у Європі), був доступний лише 1 регіон - us-east-1
3. Обмеження по кількості створених індексів та кількості вектоів у індексі

Якби датасет був 10 мільйонів статтів я б обрав один з двох варіантів: 
- Pinecone sereverless - якщо б у мене не було часу для налаштування інфраструктури і не було жорсткого complience по збереженню даних поза організацією
- Qdrant - якщо б був час на налаштування інфраструктури, постійно росло навантаження і потребувало займатись низькорівневою оптимізацією чи був жорсткий complience і дані було б заборонено передавати поза кластерами організації.
