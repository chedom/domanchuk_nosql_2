### 1. Чим Pinecone відрізняється від Qdrant і Chroma за моделлю розгортання, ліцензією і продуктивністю? У якому сценарії ви б обрали кожен із них?

Pinecone - serverless рішення, тобто, VDB розгортається у cloud, не потребує налаштування інфраструктури, інфраструктуру налаштовується, мастабується, підтримується провайдером. Зручно для старту, або якщо немає виділеного infra engineer або devops (що типово для стартапів). Мінус - дані зберігаются не на контрольованих нами серверах, може порушувати  complience (особливо критичо для фінансового сектору), менше можливості тюнингувати низькорівневі параметри індексу. Вартість виростає при збільшенні даних, запитів. Найкраще підходить для стартапів або початкових тестувань концептів.

Qdrant - self hosted рішення (хоча є і cloud опція), що потребує додаткових operation витрат (зазвичай потребує щоб у команді був infra/devops engineer), так як рішення self hosted - повний контроль над даними, гнучкість налаштування параметрів індексу, реалізований гібридний пошук. Найкаще підходить для середніх і великих компаній, де proof of concept уже протестований і є достатньо ресурсів для підтримки інфраструктури.

Chroma - self histed рішення (є опція запускати як бібліотеку), найкраще підходить для локального тестування (моментальне тествання) і навчання, не призначена для високого навантаження чи великих обємів.

### 2. Чому для задачі пошуку по науковим текстам обрана модель specter2_base, а не універсальна all-MiniLM-L6-v2? Знайдіть картку моделі на HuggingFace і процитуйте, для яких задач вона навчена.

> SPECTER2 is the successor to SPECTER and is capable of generating task specific embeddings for scientific tasks when paired with adapters. This is the base model to be used along with the adapters. Given the combination of title and abstract of a scientific paper or a short texual query, the model can be used to generate effective embeddings to be used in downstream applications.

ll-MiniLM-L6-v2 - універсальна модель для коротких тестів, specter2_base - спеціалізована для наукових статтів, розуміє близкість статтів, коли автор статті посилається на іншу наукову статтю, що є типовим для наукових статтей. Тобто, all-MiniLM-L6-v2 працює в цілому добре для текстів, проте, specter2_base натренована саме на наукових статтях і краще розуміє не просто семантику, але посилання з однієї статті на іншу, що корисно при пошуці.

### 3. Що написано у картці моделі про рекомендовану метрику схожості? Чому це важливо при створенні індексу?

У самій картці не вказано рекомендовану метику схожості. Проте, у статті, яка згадується у картці "A Multi-Format Benchmark for Scientific Document
Representations" (https://www.semanticscholar.org/reader/0d68e244faa9a5944228b571d91bc7bed88083cd) вказується, що використовувалась косинусна схожість. 
Метрика порівняння залежить від моделі, тому що, модель при навчанні будує та опитмізує векторний простір ід конкретну метрику порівняння. Тому, якщо метрики для порівняння яка використовується не співпадає тій, з якою навчалась модель - ми отримаємо не вірний результат.

### 4. Поясніть, чому при використанні нормалізованих ембеддингів (одиничної довжини) косинусна схожість (cosine similarity) еквівалентна скалярному добутку (dot product)?

Косинусна схожість згідно формули це - скалярний добуток розділений на добуток норм векторів: 
$$\text{Cosine Similarity}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \|\mathbf{v}\|}$$
У випадку нормалізації векторів - усі довжини векторів дорівнюють одиниці, тому, при нормалізованих векторах формула спрощується до скалярного добутку.
